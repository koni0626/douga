# コーディング規約・品質基準

## 1. 目的

本規約は、機能追加を続けてもソースコードの保守性、安全性、検証可能性を維持するための共通ルールを定める。

対象は次のコードと設定である。

- React / TypeScriptフロントエンド
- FastAPI / Pythonバックエンド
- バックグラウンドワーカー
- SQLAlchemyモデル、Repository、Alembic migration
- FFmpeg、OpenAI API、オブジェクトストレージなどの外部連携
- テスト、CI、デプロイ設定

本文中の用語は次の意味で使用する。

- **必須**: 原則として例外なく守る。例外時は理由を記録する。
- **推奨**: 特別な理由がなければ守る。
- **禁止**: 採用しない。

## 2. 基本原則

1. 動くことだけでなく、変更しやすく安全であることを完成条件とする。
2. 認証・認可、ユーザーデータ分離、秘密情報保護を機能要件として扱う。
3. 新規機能と不具合修正には、変更内容を検証する自動テストを追加する。
4. 小さな責務を持つモジュールを組み合わせ、巨大なファイルや万能クラスを作らない。
5. 重複、不要コード、曖昧な命名を見つけた場合、変更範囲内で改善する。
6. リファクタリングと機能変更を可能な範囲で分け、レビュー可能な差分にする。
7. 入力値、外部API、ファイル、ジョブは失敗する前提で設計する。

## 3. アーキテクチャ規約

### 3.1 バックエンドの依存方向

```text
Controller
    -> Service
        -> Repository / Unit of Work
            -> SQLAlchemy / PostgreSQL
```

依存方向を逆転させない。

### 3.2 Controller層

ControllerはFastAPIの`APIRouter`として、次だけを担当する。

- URL、HTTP Method、Status Code
- Pydanticによるリクエスト・レスポンス変換
- 認証済みユーザー、Serviceの依存注入
- Serviceの結果をHTTPレスポンスへ変換
- アプリケーション例外をHTTPエラーへ変換

Controllerでは次を禁止する。

- SQLAlchemyモデル、Session、Repositoryの直接操作
- SQLの発行
- `commit()`、`rollback()`
- 所有権、利用上限などのビジネスルール実装
- OpenAI API、FFmpeg、S3などの直接呼び出し
- 長時間処理の実行

### 3.3 Service層

Serviceはユースケース単位で実装する。

- ビジネスルールと入力条件の検証
- 認証済みユーザーに基づく所有権確認
- Repository、外部サービス、ジョブの調整
- Unit of Workを通したトランザクション境界
- ドメイン例外の送出

ServiceはFastAPIのRequest、Response、HTTP Status Codeへ依存してはならない。

### 3.4 Repository層

- SQLAlchemyによる検索と永続化をRepository内へ閉じ込める。
- ユーザー所有データの取得には`user_id`を必須引数とする。
- Repositoryは原則として`commit()`しない。
- 必要なID確定や制約検査には`flush()`を使用できる。
- 生SQLが必要な場合は、パラメータ化してRepository内に限定する。
- ControllerへORMモデルを返さず、ServiceまたはSchemaを経由する。

### 3.5 データモデル

- SQLAlchemy ORMモデルとPydantic API Schemaを分離する。
- パスワードハッシュ、セッションID、APIキーをレスポンスSchemaへ含めない。
- 日時は原則としてUTCで保存し、タイムゾーン情報を失わない。
- 主キーはUUIDを基本とする。
- ユーザー所有テーブルは`user_id`を持ち、適切な外部キーとインデックスを設定する。
- JSONBへ保存するプロジェクトデータには`schema_version`を持たせる。

### 3.6 フロントエンド

- 表示コンポーネントと、API通信・状態管理・ユースケース処理を分離する。
- APIレスポンスを無条件に信用せず、想定外状態を扱う。
- 認証状態だけを根拠に機密データを保護しない。最終的な認可は必ずバックエンドで行う。
- 共通のシーン描画ロジックを`scene-renderer`へ置き、編集画面だけに描画処理を埋め込まない。
- コンポーネントが肥大化した場合、表示、操作、データ取得、状態管理の単位で分割する。

## 4. 命名・記述規約

### 4.1 共通

- 名前から役割と単位が分かるようにする。
- `data`、`info`、`temp`、`manager`、`helper`など広すぎる名前を避ける。
- 真偽値は`is_`、`has_`、`can_`、`should_`などで始める。
- 時間には単位を含める。例: `duration_ms`、`timeout_seconds`。
- IDには対象を含める。例: `project_id`、`asset_id`。
- コメントは処理内容の読み上げではなく、理由、制約、判断を説明する。
- コメントアウトしたコードを残さない。履歴はバージョン管理へ任せる。

### 4.2 Python

- 公開関数、Service、Repositoryには型注釈を付ける。
- 非同期I/Oを行う関数は`async def`とし、同期I/Oでイベントループを塞がない。
- 可変なデフォルト引数を使用しない。
- 広すぎる`except Exception`でエラーを握り潰さない。
- 例外は意味のあるアプリケーション例外へ変換し、原因を保持する。

### 4.3 TypeScript

- `any`の使用を原則禁止し、必要な場合は理由をコメントまたはレビュー説明へ残す。
- `unknown`を受け取り、型ガードで検証する。
- API型、プロジェクトSchema、レンダラー型を重複定義しない。
- 副作用を持つ処理と純粋な変換処理を分離する。
- `dangerouslySetInnerHTML`を原則禁止する。

### 4.4 i18n

- 利用者に表示する固定UI文字列をReactコンポーネントへ直接記述しない。
- 初期対応ロケールを`ja`と`en`に限定し、既定値とフォールバックを`ja`にする。
- UIロケールとプロジェクトのコンテンツロケールを混同しない。
- 翻訳キーは`editor.caption.save`のような意味ベースの名前にし、日本語文や英語文そのものをキーにしない。
- 翻訳リソースはFeature別namespaceへ分割し、すべてを`common`へ集約しない。
- `ja`と`en`で同じキー構造を維持する。
- 単語や文節をコード上で連結して文章を作らない。文全体を翻訳単位にする。
- 動的な値には名前付き補間を使用し、件数にはi18nextの複数形機能を使用する。
- 日時、数値、割合、ファイルサイズは`Intl`などのロケール対応APIで整形する。
- ボタン幅やレイアウトを日本語の文字幅だけに合わせない。英語で長くなっても破綻しないようにする。
- バックエンドは翻訳済みエラー文をAPI契約にせず、安定した`error_code`とパラメータを返す。
- ログ、例外詳細、開発者向けメッセージは翻訳対象外とする。
- ユーザー入力の台本、プロジェクト名、素材名を自動翻訳しない。

## 5. リファクタリング規約

### 5.1 継続的な改善

- 変更した周辺コードを、変更前より悪い状態にしない。
- 機能追加時に発見した小さな重複、誤解を招く命名、不要分岐は、影響範囲が明確なら同時に改善する。
- 広範囲のリファクタリングは機能変更と分離し、先に既存挙動を固定するテストを追加する。
- 不要になったコード、Schema、migration補助処理、feature flagは削除する。
- `TODO`、`FIXME`を無期限に残さない。残す場合は理由と追跡先を記載する。

### 5.2 定期リファクタリング

次のタイミングで、専用のリファクタリング確認を行う。

- 各開発Phaseまたはマイルストーンの完了時
- 大きな機能を5～10件追加した時点
- 同じ不具合原因が2回以上発生した時点
- 下記の肥大化シグナルを検出した時点

確認結果は、即時修正、次Phase前に修正、技術的負債として追跡、のいずれかへ分類する。

### 5.3 肥大化シグナル

次は機械的な失敗条件ではないが、分割または設計見直しを必須で検討する目安とする。

- 1ファイルが400行を超える
- 1クラスが200行を超える
- 1関数が50行を超える
- ネストが4段を超える
- 1つのServiceが複数の独立したユースケースを扱う
- 同じロジックが3か所以上に複製される
- 変更のたびに無関係なテストやモジュールが壊れる
- ファイル名やクラス名に`Manager`、`Utils`、`Common`が増える
- モックしないと単体テストできない依存が多すぎる

目安を超えるコードを残す場合は、分割しない理由をレビューで説明する。

### 5.4 リファクタリング手順

1. 現在の挙動をテストで固定する。
2. 変更目的と対象外を明確にする。
3. 小さなコミットまたはレビュー可能な差分に分割する。
4. 外部仕様を変えずに内部構造を改善する。
5. ユニットテスト、統合テスト、型検査、Lintを実行する。
6. 不要になったコードとテストを整理する。

## 6. セキュリティ規約

### 6.1 基本姿勢

- セキュリティをリリース直前の確認作業にしない。
- 新機能の設計時に、入力、権限、機密性、外部通信、リソース消費を確認する。
- フロントエンドの非表示やボタン無効化を認可として扱わない。
- エラーメッセージ、ログ、レスポンスから内部情報を漏らさない。

### 6.2 認証・セッション

- パスワードはArgon2idでハッシュ化し、平文または復号可能形式で保存しない。
- セッションCookieに`HttpOnly`、`Secure`、適切な`SameSite`を設定する。
- 状態変更APIへCSRF対策を適用する。
- 登録、ログイン、パスワード関連APIへレート制限を設定する。
- ログインエラーから、メールアドレスの登録有無を推測できないようにする。
- セッション失効、ログアウト、アカウント無効化を実装する。

### 6.3 認可・ユーザーデータ分離

- 所有データへのアクセスは、すべて認証済みユーザーIDで絞り込む。
- クライアントから受け取った`user_id`を認可判断に使わない。
- UUIDが推測困難でも、所有者確認を省略しない。
- ダウンロードURLは認可後に短時間の署名付きURLとして発行する。
- ユーザーAがユーザーBのIDを指定するテストを必ず設ける。
- 将来のシステム素材は読み取り専用とし、プライベート素材と更新権限を分離する。

### 6.4 入力・出力

- API入力はPydantic Schemaで型、長さ、範囲、列挙値を検証する。
- 文字列をSQLへ連結せず、SQLAlchemyまたはパラメータ化クエリを使用する。
- Reactでは既定のエスケープを維持し、生HTMLを描画しない。
- URL入力を受け付ける機能では、許可スキーム、ホスト、リダイレクト、プライベートIPを検証し、SSRFを防ぐ。
- エラー時にスタックトレースやSQLをAPIレスポンスへ返さない。

### 6.5 ファイルアップロード

- ファイルサイズ、拡張子、MIMEタイプ、実体形式を検証する。
- 元のファイル名をストレージキーやローカルパスとして使用しない。
- アップロードをユーザー単位の領域へ保存する。
- 画像、動画、音声の解析は権限とリソースを制限したワーカーで行う。
- 不正ファイルを通常素材として公開しない。
- 必要に応じてマルウェアスキャンを追加できる経路を確保する。

### 6.6 FFmpeg・外部プロセス

- FFmpegコマンドをshell文字列として組み立てない。
- 検証済みの引数配列を使い、利用者入力をそのままオプション名へ使用しない。
- CPU、メモリ、ディスク、実行時間、出力サイズに上限を設ける。
- ワーカーをAPIプロセスおよび他ユーザーのジョブから隔離する。
- 使用するFFmpegイメージとビルド設定を固定し、セキュリティ更新を追跡する。

### 6.7 OpenAI API・外部API

- APIキーをブラウザ、DB、プロジェクトJSON、ログへ出力しない。
- Secret Managerまたはサーバー環境変数から取得する。
- ユーザー単位の生成回数、同時実行数、利用量上限を設ける。
- 429、5xxだけを制限付きで再試行し、利用者入力エラーは自動再試行しない。
- OpenAIへ送信するプロンプトと画像をUI上で明確にする。
- 外部APIレスポンスを信頼せず、形式とサイズを検証する。

### 6.8 秘密情報・ログ

- `.env`、秘密鍵、APIキーをGitへ登録しない。
- パスワード、Cookie、セッションID、Authorizationヘッダー、署名付きURLをログへ記録しない。
- メールアドレスなどの個人情報は、必要な場合だけマスクして記録する。
- ログへユーザー入力を出す場合は、改行や制御文字によるログ汚染を防ぐ。
- 本番ログの閲覧権限と保存期間を設定する。

### 6.9 依存関係

- 依存パッケージをロックファイルで固定する。
- 自動依存更新または定期的な脆弱性確認を行う。
- Pythonは`pip-audit`など、Node.jsは利用するパッケージマネージャーの監査機能をCIで実行する。
- 重大な脆弱性は通常の機能開発より優先して評価する。
- 使用していない依存パッケージを削除する。

## 7. テスト規約

### 7.1 必須ルール

- 新規ビジネスロジックにはユニットテストを追加する。
- 不具合修正では、修正前に失敗し修正後に成功する回帰テストを追加する。
- 認証、認可、ユーザーデータ分離には正常系と拒否系の両方を用意する。
- 時刻、乱数、UUID、外部APIは注入可能にし、テストを決定的にする。
- テスト間で状態を共有しない。
- 外部ネットワークへ接続するテストを通常のユニットテストに含めない。
- CIでテストが失敗した変更はマージしない。
- UI文字列を追加・変更した場合は`ja`と`en`を同時に更新する。

### 7.2 テスト分類

| 種別 | 主な対象 | 外部依存 |
| --- | --- | --- |
| Unit | Service、台本分割、時間計算、値オブジェクト | モックまたはFake |
| Controller | HTTP、Schema、Status Code、例外変換 | ServiceをFake化 |
| Repository Integration | SQLAlchemyクエリ、制約、所有者条件 | テスト用PostgreSQL |
| Migration | Alembic upgrade、データ移行 | テスト用PostgreSQL |
| Worker Integration | ジョブ状態遷移、FFmpeg引数生成 | 一部Fake、限定的に実プロセス |
| Frontend Unit | コンポーネント、Hook、状態管理 | APIをMock Service Worker等で代替 |
| E2E | 登録、ログイン、編集、書き出し要求 | 統合テスト環境 |
| Render Regression | テロップ改行、指定時刻の描画 | 固定フォント・固定Renderer |
| i18n | 翻訳キー差分、フォールバック、言語切り替え | `ja` / `en`リソース |

### 7.3 ユニットテスト対象

最低限、次のロジックをユニットテストする。

- 台本のダイアログ分割
- 日本語テロップのページ分割
- 表示時間計算と句読点待機
- 手動時間が自動再計算で失われないこと
- シーン時間の合計
- 所有権検証
- 利用量・同時実行上限
- レンダリングジョブの状態遷移
- OpenAI APIの再試行判定
- ファイル名、形式、容量の検証
- Serviceのトランザクション成功・失敗
- UIロケールの保存と`ja`へのフォールバック
- UIロケールとコンテンツロケールの独立性

### 7.4 テスト構造

- テスト名から前提、操作、期待結果が分かるようにする。
- Arrange / Act / Assertを意識する。
- 1テストでは原則として1つの振る舞いを検証する。
- 実装詳細ではなく、公開された振る舞いを検証する。
- モックの呼び出し回数だけでなく、返り値と状態変化を確認する。
- テスト用Builder、Factory、Fixtureを共通化し、巨大なセットアップを避ける。

### 7.5 カバレッジ

- 初期目標として全体のline coverage 80%以上を維持する。
- `services/`、台本・時間計算などのドメインロジックは90%以上を目標とする。
- 認証、認可、課金・利用量制限などの重要分岐は、数値目標に関係なく全経路をテストする。
- カバレッジを上げるだけの意味のないテストを追加しない。
- カバレッジ低下を伴う変更は理由をレビューで説明する。

## 8. データベース・migration規約

- SQLAlchemy 2.x形式の型付きモデルと`select()`を使用する。
- SessionまたはAsyncSessionはリクエスト・ジョブごとに生成し、並行タスクで共有しない。
- トランザクション境界はService / Unit of Workに置く。
- DBスキーマ変更はAlembic revisionを必須とする。
- 本番で`Base.metadata.create_all()`を使用しない。
- `--autogenerate`したmigrationを必ずレビューする。
- migrationは単一のデプロイジョブから適用する。
- 破壊的変更はexpand / migrate / contractへ分割する。
- migrationには、可能な範囲でdowngradeまたはロールバック方針を記載する。
- 本番データ量を考慮し、大規模なロックを発生させるDDLを無計画に実行しない。

## 9. エラー処理・可観測性

- 例外を握り潰さない。
- 利用者向けメッセージと運営者向け詳細ログを分離する。
- APIエラーには安定したアプリケーションエラーコードを付ける。
- リクエストID、ジョブID、ユーザーIDの安全な内部識別子で処理を追跡できるようにする。
- 外部APIとFFmpegの失敗には、再試行可能かどうかを明示する。
- 監視対象として、APIエラー率、ログイン失敗、ジョブ待機時間、レンダリング失敗率、OpenAI利用量を持つ。
- ユーザーへ内部スタックトレースを表示しない。

## 10. 自動チェック

実装開始後、CIで次を実行する。

### Python

- RuffによるLint・format確認
- mypyまたはPyrightによる型検査
- pytest
- coverage測定
- Alembic migration検証
- 依存関係の脆弱性監査

### TypeScript

- ESLint
- Prettier確認
- `tsc --noEmit`
- Vitest
- React Testing Library
- 日英翻訳キーの一致検査
- 依存関係の脆弱性監査

### 統合

- PostgreSQLを使用したRepositoryテスト
- 空DBへの`alembic upgrade head`
- ユーザー間アクセス分離テスト
- 主要フローのPlaywright E2E
- 固定サンプルからのレンダリングテスト

## 11. レビュー規約

- 1変更は1つの明確な目的を持つ。
- 大きな変更は、Schema、バックエンド、フロントエンド、migrationなどレビュー可能な単位へ分ける。
- 自動生成ファイルと手書きコードを同じ差分へ大量に混在させない。
- セキュリティに影響する変更は、その脅威と対策を説明する。
- DB変更にはmigrationとロールアウト順序を記載する。
- API変更にはリクエスト・レスポンス例と互換性を記載する。
- リファクタリングでは、外部挙動を変えないことをテストで示す。

## 12. Definition of Done

機能または修正は、次を満たして完了とする。

- [ ] Controller、Service、Repositoryの責務を守っている
- [ ] ユーザー所有データに認証・認可が適用されている
- [ ] 入力値と外部レスポンスを検証している
- [ ] 秘密情報や個人情報をログへ出していない
- [ ] 新規ロジックのユニットテストを追加した
- [ ] 不具合修正に回帰テストを追加した
- [ ] 正常系、異常系、権限拒否を検証した
- [ ] Lint、format、型検査、テストが成功する

## 13. 例外手続き

規約を守れない正当な理由がある場合は、黙って回避しない。変更説明またはコードコメントへ次を記録する。

1. 対象となる規約
2. 守れない理由
3. リスク
4. 代替対策
5. 解消予定または再確認条件

緊急修正で一時的に規約を満たせない場合も、回帰テストと技術的負債の追跡を後続作業として残す。
