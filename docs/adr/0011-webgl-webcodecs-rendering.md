# ADR-0011: WebGL2描画とWebCodecs書き出しを試験導入する

- Status: Accepted (Experimental)
- Date: 2026-07-20

## Context

従来の最終書き出しは、React/SVGで各フレームを描画し、PlaywrightでPNGを取得してFFmpegへ渡す。表示品質は安定しているが、ブラウザとのプロセス間通信、PNG圧縮、PNG展開がフレームごとに発生するため長尺動画では遅い。

FFmpegフィルターへ全描画仕様を移植すると高速化は期待できる一方、プレビューと書き出しの実装が分離し、テキスト、キーフレーム、カメラ効果などを二重実装する必要がある。

## Decision

共通の`WebGlProjectRenderer`を追加し、編集画面のプレビューとブラウザ内書き出しの両方から使用する。書き出しはWebCodecsでH.264/AACをエンコードし、MP4コンテナへの格納にはMediabunnyを使用する。

初期段階では次を維持する。

- ブラウザ内書き出しは「試験版」として明示する。
- 従来のサーバーFFmpeg書き出しをフォールバックとして残す。
- WebGL2、WebCodecs、H.264またはAACエンコーダーを利用できない場合はブラウザ内書き出しを開始しない。
- WebGL2が使えない環境の編集プレビューは従来SVGへ戻す。
- Project Documentと既存の時間解決関数を描画の入力とし、プレビュー専用・書き出し専用の状態を作らない。
- File System Access APIが使える環境では、完成MP4をメモリへ保持せず選択されたファイルへ逐次書き込む。
- 音声は全素材を一括デコードせず、5秒チャンクで必要な素材だけを読み込む。

## Consequences

- プレビューと試験版書き出しが同じWebGL描画器を使用するため、見た目の差を減らせる。
- PNGの生成・転送・展開がなくなり、映像フレームをGPU描画結果から直接エンコーダーへ渡せる。
- エンコード性能と対応コーデックは利用者のブラウザと端末に依存する。
- ファイルへの逐次書き込みを利用できないブラウザでは完成MP4をメモリ上に保持するため、長尺・高解像度ではメモリ上限へ達する可能性がある。
- SVGとWebGLの描画差分を継続的に検証する必要がある。
- サーバー書き出しと静止区間最適化の設計は廃止せず、互換性重視の経路として維持する。
