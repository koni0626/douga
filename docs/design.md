## 2. プロダクト概要

本アプリは、画像を中心としたスライド形式の映像に、ノベルゲームのように自動送りされるテロップ、ナレーション、BGMを組み合わせてMP4動画を制作するWebアプリである。

一般的な動画編集ソフトのように、タイムライン上で大量のテキストボックスを個別配置する操作は主目的としない。背景やレイアウトを「シーン」として作成し、長い台本を入力すると、固定されたテロップ枠へ自動改行・自動分割・自動タイミング設定して表示することを中心機能とする。

OpenAIの画像生成APIを統合し、アプリ内からシーン背景や挿絵を生成して、そのままユーザー専用の素材ライブラリへ登録できるようにする。利用者はメールアドレスとパスワードでアカウントを作成し、プロジェクト、素材、完成動画をアカウント単位で管理する。

### 2.1 プロダクト定義

> ユーザーごとに背景画像やAI生成画像を管理し、台本を入力するとノベルゲーム風のテロップ付き動画を自動構成し、Web上でMP4の書き出し履歴を管理できるアプリ。

### 2.2 想定利用者

- 画像主体の解説動画、物語動画、紹介動画を制作する個人
- CanvaやPowerPointに近い編集方法を好む利用者
- テロップの作成、配置、サイズ調整を繰り返す作業を減らしたい利用者
- サブスクリプション型の動画編集ソフトを避け、自分のPC上で制作物を管理したい利用者

## 3. 設計方針

### 3.1 台本駆動

テロップはキャンバス上の自由配置テキストではなく、時間軸を持つ「台本トラック」として管理する。利用者は文章の入力に集中し、改行、ページ分割、表示時間はアプリが初期値を生成する。

### 3.2 シーンと台本の分離

背景画像や装飾を変更する単位を「シーン」、テロップが切り替わる単位を「ダイアログ」とする。同じシーンを表示したまま、複数のダイアログを順番に送れるようにする。

### 3.3 アカウント単位のデータ分離

プロジェクト、素材、生成画像、完成動画は、認証済みユーザーの所有物としてサーバー側で管理する。すべての読み取り・更新・削除で、認証中ユーザーが対象データの所有者であることをバックエンドが検証する。クライアントから送られた`user_id`をアクセス制御の根拠にしない。

### 3.4 非破壊編集

元の素材ファイルは変更しない。編集内容はプロジェクトデータとして保存し、完成動画はレンダリングによって生成する。

### 3.5 同一描画系の利用

プレビューと最終書き出しで、共通のTypeScript描画パッケージとシーン定義を使用する。ブラウザ上のプレビューと、サーバー上のヘッドレスブラウザによるフレーム描画で同じレイアウトロジックを動かし、テロップの折り返し、座標、フォントサイズが変わらないことを重視する。

### 3.6 非同期処理

画像生成、素材解析、サムネイル生成、動画レンダリングは同期HTTPリクエスト内で完了させない。ジョブキューへ登録し、ワーカーで処理して、画面へ状態と進捗を通知する。

### 3.7 将来公開を前提とした拡張性

初期段階でも、認証、所有者ID、オブジェクトストレージ、ジョブ分離を省略しない。全ユーザー共通素材、メール認証、課金、利用量制限は将来機能とするが、データモデルには追加可能な拡張点を用意する。

### 3.8 i18n前提

UI文字列をコンポーネントへ直接埋め込まず、初期リリースから国際化基盤を使用する。対応言語は日本語`ja`と英語`en`とし、未設定時と翻訳欠落時のフォールバックは日本語`ja`とする。

アプリのUI言語と、制作する動画・台本の本文言語は別の値として管理する。日本語UIで英語動画を作る場合や、英語UIで日本語動画を作る場合に対応する。

## 6. 主要利用フロー

### 6.1 ユーザー登録とログイン

1. 利用者がメールアドレスとパスワードを入力する。
2. FastAPIがメールアドレスの形式と重複を検証する。
3. パスワードをArgon2idでハッシュ化し、ユーザーを作成する。
4. UIロケールを明示指定がなければ`ja`で保存する。
5. 初期バージョンではメール確認を行わず、そのままログイン可能にする。
6. ログイン成功時にサーバー側セッションを作成する。
7. ブラウザにはセッション識別子を`HttpOnly`、`Secure`、適切な`SameSite`属性を持つCookieとして保存する。
8. 以後のAPI呼び出しでは、Cookieからユーザーを特定する。

### 6.2 新規動画の制作

1. プロジェクトを作成する。
2. 出力解像度、FPS、テロップスタイルを選択する。
3. 画像を読み込むか、OpenAI APIで背景画像を生成する。
4. 背景画像を使用してシーンを作成する。
5. シーンへ台本を入力または貼り付ける。
6. アプリが台本をテロップ枠に合わせてページ分割する。
7. アプリが各ページの表示時間を自動計算する。
8. 必要に応じて文章、分割位置、表示時間を調整する。
9. ナレーションとBGMを追加する。
10. プレビューを確認する。
11. MP4を書き出す。
12. 完成動画をユーザーの「書き出し履歴」へ登録する。

### 6.3 AI画像の生成

1. 素材ライブラリで「画像生成」を選択する。
2. プロンプト、サイズ、品質を指定する。
3. バックグラウンドジョブとして生成を開始する。
4. 生成途中、成功、失敗の状態を表示する。
5. 成功時に画像ファイルと生成メタデータを素材ライブラリへ登録する。
6. シーン背景または画像レイヤーとして使用する。

### 6.4 台本の自動送り

1. 利用者が複数段落の台本を入力する。
2. 段落または明示的な区切りからダイアログを作成する。
3. テロップ枠へ収まらないダイアログを複数ページへ分割する。
4. 各ページへ自動表示時間を設定する。
5. シーン内でページを順番に表示する。
6. ページの合計時間からシーン時間を決定する。

### 7.4 i18n・言語設定

フロントエンドには`react-i18next`を使用し、翻訳リソースを機能ごとのnamespaceへ分割する。

初期設定:

| 項目 | 値 |
| --- | --- |
| 対応UIロケール | `ja`、`en` |
| 既定UIロケール | `ja` |
| フォールバック | `ja` |
| 初回未ログイン画面 | `ja` |
| ログイン後 | `users.preferred_locale` |

言語切り替えは設定画面とログイン画面から行えるようにする。ログイン前の選択はブラウザへ保存し、ログイン後にユーザー設定へ反映できるようにする。ただしブラウザ言語からの自動切り替えは行わず、既定は常に日本語とする。

バックエンドは人間向けの翻訳済みエラー文を主要契約にせず、安定した`error_code`と補間用パラメータを返す。Reactが`error_code`を翻訳キーへ対応させる。ログと開発者向け詳細は翻訳対象外とする。

日時、数値、ファイルサイズは`Intl.DateTimeFormat`、`Intl.NumberFormat`など、ロケール対応APIで表示する。単語を文字列連結して文章を組み立てず、複数形と補間を翻訳リソース側で扱う。

参考資料:

- [i18next fallback](https://www.i18next.com/principles/fallback)
- [i18next plurals](https://www.i18next.com/translation-function/plurals)

## 8. ユーザー・認証・認可

### 8.1 MVPの認証方式

- メールアドレスとパスワードで登録する。
- メールアドレスはログイン識別子として重複を許可しない。
- パスワードは平文または復号可能な暗号文で保存せず、Argon2idでハッシュ化する。
- 認証状態はサーバー側セッションとして管理する。
- セッションCookieはJavaScriptから読み取れない`HttpOnly`を使用する。
- 状態変更APIにはCSRF対策を適用する。
- 登録・ログインAPIにIPおよびアカウント単位のレート制限を設ける。
- ログインエラーは、メールアドレスの存在有無を推測できない共通メッセージにする。

パスワード保存の指針として、[OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)を参照する。

### 8.2 将来のメール確認

要望として「登録時にメールで仮パスワードを送る」方式があるが、実装時はパスワード自体をメール送信しない。次の方式を推奨する。

1. 利用者がメールアドレスを登録する。
2. 有効期限付き・一度だけ使用可能な確認トークンをメール送信する。
3. 利用者が確認リンクを開く。
4. 利用者自身がパスワードを設定する。
5. トークンを無効化し、登録を完了する。

トークン方式の詳細は[OWASP Forgot Password Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html)を参照する。MVPではメール送信基盤を実装せず、ユーザー作成直後からログイン可能とする。

### 8.3 認可とテナント分離

- `projects.user_id`、`assets.user_id`、`exports.user_id`を必須とする。
- APIの検索条件には、認証済みユーザーIDを必ずバックエンド側で追加する。
- IDだけで対象を取得してから所有者を確認する実装を避ける。
- オブジェクトストレージのキーは`users/{user_id}/...`で分離する。
- ダウンロードは認可確認後に、短時間だけ有効な署名付きURLを発行する。
- レンダリングワーカーも、ジョブに記録された所有者と各素材の所有者を検証する。
- 管理者権限とシステム素材は将来追加し、通常ユーザーの所有権と混在させない。

### 8.4 将来のシステム素材

将来、全ユーザーへ提供する汎用素材を追加できるよう、素材には`scope`を持たせる。

```text
private: 所有ユーザーのみ利用可能
system:  全ユーザーが読み取り可能、運営者のみ更新可能
```

MVPでは`private`のみ実装し、一般ユーザー同士の素材共有は行わない。

### 8.5 認証API案

| Method | Path | 用途 |
| --- | --- | --- |
| POST | `/api/auth/register` | メールアドレス・パスワードで登録 |
| POST | `/api/auth/login` | ログインしてセッションを作成 |
| POST | `/api/auth/logout` | 現在のセッションを破棄 |
| GET | `/api/auth/me` | 現在のユーザーを取得 |
| DELETE | `/api/auth/sessions` | 将来、他端末を含む全セッションを破棄 |

ReactとFastAPIは、可能であれば同一サイトの`/`と`/api`として公開し、Cookie、CORS、CSRFの構成を単純化する。

## 9. テロップ・台本仕様

### 9.1 テロップ枠

テロップ枠はプロジェクト全体またはテンプレート単位で定義する。シーンやダイアログごとにテキストボックスを作成しない。

主な設定項目は次のとおり。

- アンカー位置: 上、中央、下、および左右
- X/Yオフセット
- 幅、高さ
- 内側余白
- フォントファミリー、ウェイト、サイズ
- 文字色、縁取り、影
- 背景色、背景画像、角丸、透明度
- 横方向・縦方向の配置
- 最大行数
- 行間、文字間隔
- 話者名の表示有無と位置

### 9.2 自動レイアウト

`DialogueLayoutEngine`が、最終レンダリングと同じフォント計測処理を使って、次の処理を行う。

1. 明示的な改行を保持する。
2. テロップ枠の有効幅に合わせて折り返す。
3. `content_locale=ja`では日本語の禁則処理、`content_locale=en`では単語境界を優先した折り返しを行う。
4. 最大行数を超える場合は複数ページへ分割する。
5. 分割結果を利用者が手動修正できるようにする。
6. スタイル変更時は自動ページ分割を再計算する。

自動縮小は既定動作にしない。ページごとに文字サイズが変わると読みにくいため、文字が収まらない場合はページ分割を優先し、分割できない長い単語などは警告を表示する。

### 9.3 表示時間の自動計算

`DialogueTimingEngine`は、ページごとに次の式を基準として表示時間を求める。

```text
表示時間 = 表示文字数 / 1秒あたりの文字数
         + 句読点による待ち時間
         + 読了後の余韻
```

初期値はコンテンツロケールごとに保持し、すべてプロジェクト設定で変更可能とする。下表は日本語`ja`の既定値であり、英語`en`は単語数を基準にした時間計算として別設定を持たせる。

| 設定 | 初期値 |
| --- | ---: |
| 読む速さ | 10文字/秒 |
| `、`による追加時間 | 0.2秒 |
| `。！？`による追加時間 | 0.5秒 |
| 最低表示時間 | 2.0秒 |
| 読了後の余韻 | 0.8秒 |
| ページ間の空白時間 | 0.0秒 |

表示時間はページ単位で手動上書きできる。自動値と手動値を区別し、台本変更時に手動値を不用意に失わないようにする。

### 9.4 表示効果

MVPでは次の効果を提供する。

- `instant`: ページ全文を即時表示
- `fade`: ページ全文をフェードイン
- `typewriter`: 文字を順番に表示

タイプライター表示では、文字表示速度とページ表示時間を別パラメータとして保持する。すべての文字を表示し終わったあと、読了待機時間を確保する。

### 9.5 ナレーションとの同期

MVPではナレーションファイルをシーンまたはダイアログへ手動で関連付ける。関連付けられた場合、次の優先順位で時間を決める。

1. 利用者が設定した手動時間
2. 関連付けられたナレーション音声の長さと前後余白
3. 文字数による自動計算

音声認識による自動同期は将来機能とする。

## 10. シーン・レイヤー仕様

### 10.1 シーン

シーンは以下を保持する。

- シーンID、名称、並び順
- 背景色または背景素材
- レイヤー一覧
- ダイアログ一覧
- シーン開始・終了トランジション
- シーン全体の音声
- サムネイル

シーンの長さは原則として、ダイアログページ、ページ間余白、関連音声の合計から自動計算する。利用者が明示的にシーン長を固定することもできる。

### 10.2 レイヤー

MVPでサポートするレイヤー種別は次のとおり。

- 画像
- 自由テキスト
- 長方形・楕円などの基本図形
- 動画

各レイヤーは位置、サイズ、回転、透明度、表示順、表示区間を保持する。テロップ本文はレイヤーではなく台本トラックとして扱う。

自由テキストレイヤーは横書き・縦書きを選択できる。テキスト固有の属性として、フォントファミリー、文字サイズ、文字色、表示方式を保持する。ネオン表示では文字色とは別にネオン色を持ち、既定値を淡い青`#9bdcff`とする。文字送りはレイヤー開始時刻と1秒あたりの表示文字数から表示中の文字列を決定し、プレビューと書き出しで同じ時刻計算を使用する。

### 10.3 トランジション

MVPではカットとクロスフェードのみを提供する。トランジション時間はシーンごとに設定できる。

## 12. OpenAI画像生成連携

### 12.1 API選択

MVPでは単発の生成と編集を中心とするため、Image APIを使用し、モデルには`gpt-image-2`を指定する。会話形式で継続編集する機能を追加する場合は、Responses APIの採用を別途検討する。

参考資料:

- [OpenAI Image generation guide](https://developers.openai.com/api/docs/guides/image-generation)
- [GPT Image 2 model](https://developers.openai.com/api/docs/models/gpt-image-2)

### 12.2 既定値

| 用途 | サイズ | 品質 | 形式 |
| --- | --- | --- | --- |
| 下書き・構図確認 | 2048x1152 | low | jpegまたはwebp |
| 通常素材 | 2048x1152 | medium | pngまたはwebp |
| 最終素材 | 2048x1152 | high | png |

16:9の動画では2048x1152で生成し、レンダリング時に1920x1080へ縮小する。テロップや正確な文字は生成画像へ描かせず、アプリ側のテキスト描画で合成する。

### 12.3 認証情報

- OpenAI APIキーはサーバー運営者が管理し、プロジェクトJSON、PostgreSQL、ブラウザ、ログへ保存しない。
- APIキーはデプロイ環境のSecret Managerまたは環境変数からFastAPI・ワーカーだけが取得する。
- ReactへAPIキーを返さず、画像生成は必ず認証済みバックエンドAPIを経由する。
- ユーザー単位の回数・生成量制限を設け、第三者によるAPI費用の濫用を防ぐ。
- 将来、ユーザー自身のAPIキーを使うBYOK方式を採用する場合は、暗号化保存とキー更新・削除手順を別途設計する。

### 12.4 ジョブ処理

画像生成はUIスレッド外で実行し、次の状態を持つ。

```text
queued -> running -> succeeded
                  -> failed
                  -> cancelled
```

一時的な通信失敗、HTTP 429、5xxは指数バックオフ付きで再試行する。プロンプトや入力画像の修正が必要な利用者エラー、コンテンツ制限によるエラーは自動再試行しない。

### 12.5 生成履歴

- 生成結果は上書きせず、新しい素材として保存する。
- プロンプトと設定を再利用できるようにする。
- 親素材IDを使って編集・バリエーションの派生関係を保持する。
- 失敗した生成ジョブも、秘密情報を除いたエラー種別と設定を履歴として残す。

## 13. 音声仕様

### 13.1 MVPで扱う音声

- ダイアログまたはシーンに関連付けたナレーション
- プロジェクト全体または区間に配置するBGM
- 任意のタイミングに配置する効果音

### 13.2 音量制御

- 素材ごとのゲイン
- フェードイン、フェードアウト
- ナレーション中のBGM音量を下げる簡易ダッキング
- プロジェクト全体のマスター音量

波形編集や高度なノイズ除去はMVP対象外とする。

### 13.3 AivisSpeechナレーション生成

- AivisSpeech Engineの`/speakers`、`/audio_query`、`/synthesis`をバックエンドから利用する。
- ブラウザからEngineへ直接接続せず、APIキーやローカルEngineの場所をクライアントへ公開しない。
- 生成結果はWAVのユーザー専用素材として保存し、既存の音声トラックへナレーション用途で配置する。
- 話者スタイル、話速、感情表現の強さ、テンポの緩急、生成音量を指定できる。
- 1回の文章は既定500文字までとする。テロップ同期時は句読点と最大文字数で短い区間へ分割して個別合成する。
- 同期用音声は区間ごとのWAVフレーム数から開始・終了時刻を実測し、素材メタデータへテロップキューとして保存する。
- AivisSpeechのモーラ長は利用せず、文字数による推測だけで同期済みとは判定しない。
- 開発環境のWindowsでは公式の標準インストール先を探索して遅延起動できる。本番環境では専用Engineまたはコンテナを明示設定する。

## 14. 書き出し・完成動画管理

### 14.1 既定出力

| 項目 | 既定値 |
| --- | --- |
| 解像度 | 1920x1080 |
| フレームレート | 10fps |
| 映像コーデック | H.264 |
| 音声コーデック | AAC |
| コンテナ | MP4 |

ユーザー設定の解像度とFPSは新規プロジェクトの初期値として扱う。既存プロジェクトでは編集画面のプロジェクト設定からFPSを変更し、自動保存によって新しいrevisionを作成する。書き出し開始時にはファイル名、解像度、FPSを確認・一時変更でき、その変更は作成するexportだけへ固定してProject Documentへは反映しない。

### 14.2 レンダリングパイプライン

```text
React Editor
    -> Project JSONをFastAPIへ保存
    -> Render Jobをキューへ登録
    -> Render WorkerがProjectと素材の所有権を再検証
    -> 共通TypeScript Rendererをヘッドレスブラウザで実行
    -> 映像フレームをFFmpegへ入力
    -> FFmpegで音声ミックス・エンコード
    -> 一時オブジェクトを検証
    -> 完成MP4をユーザー領域へ確定保存
    -> Export Historyへ登録
```

FFmpegはテロップ編集UIそのものではなく、最終的な映像エンコード、音声合成、形式変換に使用する。テロップとレイヤーの描画はブラウザ互換の共通レンダラーが担当し、Reactのプレビューとサーバー書き出しの差異を抑える。デバッグ用として、指定時刻のフレームをPNGへ書き出す機能も用意する。

画像素材はBase64へ変換してProject JSONへ埋め込まない。ワーカーが所有権と保存先を検証したローカルファイルだけをレンダラー専用URLへ対応付け、ヘッドレスブラウザから要求された素材をPlaywrightが応答する。ブラウザへ渡すProject JSONには素材IDと一時URLだけを含め、ローカルファイルパスはページへ公開しない。

レンダリングワーカーはFastAPIのWebプロセスから分離し、CPU・メモリ・実行時間・同時実行数を制限する。FFmpegコマンドへ利用者入力を文字列連結せず、検証済み引数配列として渡す。

### 14.3 書き出し履歴

書き出しのたびに新しいバージョンを作成し、既存MP4を暗黙に上書きしない。

保持する情報:

- 書き出しID、プロジェクトID
- バージョン番号
- ファイルパス、ファイルサイズ
- 作成日時
- 動画時間
- 解像度、FPS、コーデック
- 書き出し時のプロジェクトリビジョン
- 成功、失敗、キャンセルの状態
- エラー概要とFFmpegログの場所

提供する操作:

- 再生
- ブラウザでダウンロード
- 同じ設定で再書き出し
- 書き出し設定を変更して再書き出し
- 古いバージョンの削除

### 14.4 FFmpegのライセンス方針

FFmpegは商用サービスでも利用可能である。ただし「商用だから有料ライセンスが必要」という形ではなく、採用するFFmpegビルドと外部ライブラリのライセンス条件を守る必要がある。

- FFmpeg本体の大部分はLGPL 2.1以降で提供される。
- `--enable-gpl`でGPL部分または`libx264`、`libx265`などのGPLライブラリを有効化すると、FFmpegビルド全体がGPL扱いになる。
- `--enable-nonfree`を有効化したビルドは再配布できない。
- SaaSとして自社サーバー上でFFmpegを実行し、バイナリをユーザーへ配布しない構成では、通常のGPL/LGPLはサーバー利用だけを理由にアプリ全体のソース公開を要求するライセンスではない。ただし、FFmpegやコンテナイメージを顧客へ配布する形へ変更した場合は、配布条件を再評価する。
- FFmpegが生成した通常の動画ファイルに、FFmpegのGPL/LGPLが自動的に適用されるわけではない。
- コーデックにはソフトウェアライセンスとは別に特許・地域別の問題があり得るため、本格的な商用提供前に利用地域とコーデック構成を専門家へ確認する。

MVPでは次の運用を採用する。

1. 使用するDockerイメージまたはFFmpegバイナリの配布元、バージョン、`ffmpeg -buildconf`を記録する。
2. `--enable-nonfree`を含むビルドを使用しない。
3. GPL構成を避けたい場合は、`--enable-gpl`、`libx264`、`libx265`を含まないLGPL構成を選定する。
4. GPL構成を採用する場合も、Webサービス内で独立したコマンドとして実行し、配布形態を変更する前に再確認する。
5. FFmpegの著作権表示、ライセンス、使用ビルド情報を運営用ドキュメントへ記録する。

参考資料:

- [FFmpeg License and Legal Considerations](https://ffmpeg.org/legal.html)
- [FFmpeg current LICENSE](https://ffmpeg.org/doxygen/trunk/md_LICENSE.html)
- [GNU GPL FAQ](https://www.gnu.org/licenses/gpl-faq.en.html)

本節は設計上の方針であり、個別事業に対する法的助言ではない。

## 17. ソフトウェア構成

```text
Browser
  React / TypeScript / Canvas or DOM Renderer
       |
       | HTTPS / JSON / Signed Upload
       v
FastAPI
  Authentication / Authorization / Project API / Asset API / Job API
       |                 |                    |
       v                 v                    v
  PostgreSQL       Object Storage         Redis Queue
                                             |
                                             v
                                      Background Workers
                              Image Generation / Media Probe / Render
                                             |
                                  Headless Browser + FFmpeg
```

FastAPIは長時間のレンダリングを直接実行せず、ジョブの登録と状態取得を担当する。ワーカーは別プロセスまたは別コンテナとして水平スケール可能にする。

FastAPI内部は次の依存方向を守る。

```text
Controller (FastAPI Router)
    -> Service (Application Use Case)
        -> Repository / Unit of Work
            -> SQLAlchemy / PostgreSQL
```

下位層から上位層へ依存しない。バックグラウンドワーカーはControllerを経由せず、Serviceまたは専用ユースケースを直接呼び出す。

### 17.1 推奨モジュール

```text

   ├─ src/douga/
   │  ├─ core/
   │  ├─ db/
   │  ├─ modules/               # 機能単位で層を分離
   │  ├─ integrations/
   │  └─ workers/
   ├─ migrations/
   └─ tests/

├─ project-schema/              # JSON Schema / TypeScript型
└─ scene-renderer/              # プレビュー・書き出し共通描画
infra/

```

詳細と依存規則は[フォルダ構成設計](folder-structure.md)を参照する。

### 17.2 技術候補

| UI | React / TypeScript |

| キャンバス | Konva.js、Fabric.js、または独自Canvas/DOM描画 |
| サーバー最終描画 | 共通TypeScriptレンダラー + Playwright/Chromium |

| バックエンドテスト | pytest / HTTPX |

### 17.3 Controller層

Controller層はFastAPIの`APIRouter`として実装し、次の責務だけを持つ。

- URL、HTTP Method、Status Codeの定義
- Pydanticによるリクエスト・レスポンス変換
- 認証済みユーザー、リクエストID、Serviceの依存注入
- Serviceが返した結果のHTTPレスポンス化
- アプリケーション例外からHTTPエラーへの変換

Controller層では、SQLAlchemyモデルの直接操作、SQL発行、`commit()`、動画生成、OpenAI API呼び出しを行わない。所有権検証などのビジネスルールもService層へ委譲する。

### 17.4 Service層

Service層は「ユーザー登録」「プロジェクト更新」「素材登録」「レンダリング開始」などのユースケース単位で実装する。

- ユースケースの入力検証
- 認証済みユーザーに基づく所有権・利用上限の検証
- 複数Repositoryや外部サービスの調整
- トランザクション境界の管理
- ドメイン例外の送出
- ジョブ登録と監査情報の記録

ServiceはHTTPのRequest、Response、Status Codeへ依存しない。Controller以外に、バックグラウンドワーカー、CLI、テストからも呼び出せる形にする。

### 17.5 Repository・Unit of Work

Repository層はSQLAlchemyを使った取得・保存を担当する。

- `UserRepository`、`ProjectRepository`、`AssetRepository`、`ExportRepository`などを定義する。
- ユーザー所有データの取得メソッドは`user_id`を必須引数にする。
- Repositoryは`commit()`しない。必要な場合だけ`flush()`を行う。
- 1ユースケースの`commit()`または`rollback()`はServiceがUnit of Workを通して決定する。
- 生SQLが必要な場合もRepository内へ閉じ込め、ControllerやServiceへSQL文字列を出さない。

SQLAlchemyの`Session`または`AsyncSession`は、HTTPリクエストまたはワーカージョブごとに1つ作成する。並行タスク間で同一Sessionを共有しない。公式ドキュメントでも、Sessionは1つのトランザクションを表す状態を持つオブジェクトであり、`AsyncSession`はタスクごとに分けることが案内されている。

参考資料: [SQLAlchemy Session Basics](https://docs.sqlalchemy.org/en/20/orm/session_basics.html)

### 17.6 Alembicマイグレーション方針

- DBスキーマの作成・変更はAlembic revisionで管理する。
- 本番起動時に`Base.metadata.create_all()`でスキーマを自動作成しない。
- SQLAlchemy metadataをAlembicへ接続し、`--autogenerate`を利用できるようにする。
- 自動生成されたmigrationは候補として扱い、型変更、制約、データ移行、downgradeを人がレビューする。
- revisionファイルをアプリケーションコードと同じリポジトリでバージョン管理する。
- migration適用は複数APIコンテナから同時実行せず、デプロイ時の単一ジョブとして`alembic upgrade head`を実行する。
- 本番で停止時間を抑える変更は、列追加、データ移行、参照切替、旧列削除を分けるexpand/contract方式を使う。
- 制約名とインデックス名にはSQLAlchemyのnaming conventionを設定し、環境ごとの差異を防ぐ。

CIでは少なくとも次を確認する。

1. 空のPostgreSQLへ`alembic upgrade head`を適用できる。
2. 1つ前のリビジョンから最新へアップグレードできる。
3. ORM metadataとmigration済みスキーマに意図しない差分がない。
4. 必須データ移行を含むmigrationテストが成功する。

Alembicの`--autogenerate`はmigration候補を生成する機能であり、生成結果の手動レビューを前提とする。

参考資料: [Alembic Autogenerate](https://alembic.sqlalchemy.org/en/latest/autogenerate.html)

## 18. Undo・Redoと自動保存

- 編集操作はコマンドとして表現し、Undo・Redoできるようにする。
- テキスト入力は一定時間内の連続入力を1操作としてまとめる。
- Undo・Redo履歴はブラウザセッション内で管理し、サーバー保存とは分離する。
- 自動保存は数秒間操作がないタイミング、または一定間隔でFastAPIへ送信する。
- プロジェクトにリビジョン番号を持たせ、古い画面からの保存で新しい内容を上書きしないよう楽観ロックを行う。
- 通信失敗中の変更はブラウザの一時領域へ保持し、再接続時に復旧を提案する。
- サーバーは一定間隔または重要操作時にプロジェクトリビジョンを保存する。
- 素材ファイルの登録・削除は、プロジェクト編集のUndoとは別のライブラリ操作として扱う。

## 19. 非機能要件

### 19.1 性能

- 100シーン、1,000ダイアログ程度で台本編集が実用的に動作すること。
- APIの通常操作は長時間処理を含めず、画像生成・解析・レンダリングをワーカーで実行すること。
- シーク時は低解像度プレビューやキャッシュを利用できる構造にすること。
- 素材サムネイルは遅延生成し、一覧表示を妨げないこと。
- レンダリング同時実行数と、ユーザーごとの待機ジョブ数を制限できること。

### 19.2 信頼性

- レンダリング失敗時もプロジェクトと既存書き出しを失わないこと。
- 一時ファイルを完成動画として扱わず、成功後に確定ファイルへ置き換えること。
- PostgreSQLのメタデータとオブジェクトストレージの不整合を検出・修復できること。
- FFmpegの終了コードとログを保存すること。
- ジョブワーカー停止後も、実行中ジョブを再試行または失敗確定できること。

### 19.3 セキュリティ・プライバシー

- パスワード、セッションID、APIキーをログへ出力しないこと。
- パスワードをArgon2idでハッシュ化し、平文・復号可能形式で保存しないこと。
- ログへAPIキー、Authorizationヘッダーを出力しないこと。
- OpenAI APIへ送信するプロンプトと画像を利用者が認識できるUIにすること。
- すべての所有データAPIで認証と所有者確認を行うこと。
- ユーザーの素材を、明示した画像生成・編集処理以外でOpenAI APIへ送信しないこと。
- アップロードファイルとFFmpeg処理を隔離し、リソース上限とタイムアウトを設けること。

### 19.4 可搬性

- プロジェクトJSONにはスキーマバージョンを含めること。
- パスは可能な範囲で素材IDを通して参照すること。
- 将来、ユーザーがプロジェクトと使用素材をZIPへエクスポートできる設計にすること。

## 20. エラー処理

| エラー | UIでの対応 |
| --- | --- |
| 登録済みメールアドレス | アカウント列挙を考慮した共通メッセージを表示 |
| ログイン失敗 | メールまたはパスワードのどちらが原因かを区別しない |
| セッション期限切れ | 編集中データを保持したまま再ログインへ誘導 |
| 権限のないデータ参照 | 内容を漏らさず404または403を返す |
| アップロード失敗 | 再送操作と容量・形式エラーを表示 |
| 素材オブジェクトが読めない | 対象素材と再アップロード・削除操作を表示 |
| FFmpegが利用できない | レンダリングジョブを失敗にし、運営ログへ詳細を保存 |
| レンダリング失敗 | 失敗地点、概要、ログを表示 |
| OpenAI APIキー未設定 | 画像生成を停止し、運営側設定エラーとして扱う |
| OpenAI認証エラー | ユーザーへ一般的な障害を表示し、運営ログへ詳細を保存 |
| OpenAIレート制限 | 待機時間を表示して再試行 |
| 生成内容が拒否された | プロンプト修正を案内し、自動再試行しない |
| ユーザー利用量上限 | 使用量と削除導線を表示 |
| サーバー保存容量不足 | 新規処理を停止し、運営監視へ通知 |

## 22. MVP受け入れ条件

以下をすべて満たした時点を、最初の利用可能なバージョンとする。

1. メールアドレスとパスワードでユーザー登録・ログイン・ログアウトできる。
2. パスワードがArgon2idで保存され、平文を復元できない。
3. ユーザーAがユーザーBのプロジェクト、素材、完成動画を参照・変更できない。
4. 利用者が16:9の背景画像をアップロードできる。
5. 利用者が固定テロップ枠の位置とスタイルを設定できる。
6. 複数段落の日本語台本を一括で貼り付けられる。
7. 台本がテロップ枠に合わせて自動改行・自動ページ分割される。
8. 各ページに自動表示時間が設定される。
9. 背景を維持したまま、ページが順番に自動送りされる。
10. 全文表示またはタイプライター表示を選択できる。
11. 分割位置と表示時間を手動調整できる。
12. ブラウザプレビューと完成動画でテロップの改行位置が一致する。
13. プロジェクトをサーバーへ保存して再度開ける。
14. 非同期ジョブとして1920x1080、30fpsのMP4を書き出せる。
15. 完成MP4を認可されたユーザーだけが再生・ダウンロードできる。
16. アプリ内からGPT Image 2で画像を生成し、背景として利用できる。
17. OpenAI APIキーがブラウザ、プロジェクト、データベース、ログへ露出しない。
18. 書き出しに失敗してもプロジェクトデータと既存の完成動画を失わない。
19. 空のPostgreSQLへAlembic migrationを適用してアプリを起動できる。
20. ControllerがSQLAlchemyモデルやSessionを直接操作していない。
21. ログイン前・新規ユーザー・翻訳欠落時のUIが日本語になる。
22. 設定画面で日本語と英語を切り替え、再ログイン後も選択が保持される。
23. 日本語と英語の翻訳キーに欠落差分がないことをCIで検出できる。
24. UIロケールとプロジェクトのコンテンツロケールを独立して設定できる。

## 23. 主なリスクと対策

| リスク | 対策 |
| --- | --- |
| 日本語の折り返し品質が低い | 独立したレイアウトエンジンとして早期に技術検証する |
| プレビューと出力で文字位置がずれる | 共通TypeScriptレンダラー、同一Webフォント、Chromiumバージョン固定を利用する |
| フレーム生成が遅い | キャッシュ、低解像度プレビュー、静止区間の再利用を導入する |
| 大量PNGによる容量消費 | 通常はraw videoをFFmpegへ直接渡し、PNGはデバッグ時のみ使用する |
| AI生成の待ち時間でUIが固まる | バックグラウンドジョブと進捗表示を使う |
| AI画像内の文字が不正確 | テロップやタイトルはアプリ側で描画する |
| 素材削除でプロジェクトが壊れる | 素材ID参照と削除前の参照確認を行う |
| ユーザー間でデータが漏れる | 全クエリの所有者条件、署名付きURL、認可の自動テストを必須化する |
| 不正アップロードがFFmpegを攻撃する | 種別・容量検証、隔離ワーカー、権限制限、タイムアウトを導入する |
| レンダリング負荷でAPIが停止する | APIとワーカーを分離し、キュー、同時実行数、CPU・メモリ制限を設ける |
| OpenAI API費用を濫用される | ユーザー別クォータ、レート制限、監視、上限到達時の停止を実装する |
| FFmpegビルドのライセンスが不明 | バージョンと`-buildconf`を固定・記録し、`nonfree`を禁止する |
| Controllerへ業務ロジックが集中する | Controllerの責務をHTTP変換に限定し、Service単体テストを必須化する |
| migrationとORMモデルが乖離する | CIで空DBへのupgradeとschema差分検査を実行する |
| 複数コンテナが同時にmigrationする | デプロイ時の単一migrationジョブだけが適用する |
| 機能範囲が汎用動画編集へ拡大する | 台本駆動型の制作フローを優先し、MVP対象外を維持する |
